Mira V3: Workflow-First Agent Platform

From monolithic super-agent to composable workflows + atomic agents + hard policies

This document defines the complete architecture for Mira V3. The core thesis: the unit of execution is the workflow, not the agent. Agents are stateless executors. Policies are executable code. Workflows connect them.

North Star: Personal assistant + content creation + research-for-content + self-evolution. Everything in this document serves that goal. If a design decision doesn't connect to one of these four, cut it.

2. The Three Primitives

Everything in Mira V3 reduces to three primitives. Skills, memory, integrations, and UI all serve these.

2.1 Workflows

A workflow is a DAG of typed steps with checkpoints, eval gates, and retry logic. Defined in Python (not YAML — conditional logic requires real code). Each step has a typed input/output contract.

@dataclass
class Step:
    name: str
    type: Literal["agent", "policy", "eval", "human", "monitor", "parallel", "deterministic"]
    agent: str | None          # which agent runs this step
    policies: list[str]        # policies checked after step completes
    eval_criteria: dict | None # for eval steps: thresholds, bars
    timeout_s: int
    retries: int
    on_fail: Literal["retry", "skip", "abort", "escalate"]
    input_schema: type         # typed dataclass
    output_schema: type        # typed dataclass

@dataclass
class Workflow:
    name: str
    trigger: Trigger           # schedule, event, manual, monitor
    steps: list[Step]
    priority: int              # for preemption
    version: int               # for safe updates
    max_duration_s: int

2.2 Agents

An agent is a stateless executor with a designated model, loaded skills, and a token budget. It receives a typed input, calls an LLM (or runs deterministic code), and returns a typed output. No side effects beyond writing to the workflow's workspace.

class Agent(Protocol):
    name: str
    model: str                # configurable via WebGUI
    skills: list[str]         # loaded into prompt context
    token_budget: int         # max tokens per invocation

    def execute(self, input: StepInput) -> StepOutput:
        """Run one workflow step. Stateless. No side effects beyond workspace."""
        ...

2.3 Policies

A policy is executable Python code (not LLM text) that validates inputs, outputs, or state. Policies have test suites. A failed policy blocks workflow progression. No exceptions, no overrides.

class Policy(Protocol):
    name: str
    category: Literal["content", "timing", "quality", "safety", "compliance"]

    def check(self, ctx: PolicyContext) -> PolicyResult:
        """Returns pass/fail with reasons. ZERO LLM calls."""
        ...

@dataclass
class PolicyResult:
    passed: bool
    reason: str
    details: dict

5. Policy Framework

Policies are executable code with test suites. A policy that can be expressed as an LLM prompt is too soft. If the rule matters, it's a function with assertions.

Hard rule: Policies are Python functions that return pass/fail. They run in the Policy Runner agent (deterministic, zero LLM calls). A failed policy BLOCKS workflow progression. The coder agent can run policy tests at any workflow step.

5.3 Policy Execution in Workflow

# How a policy step runs in the workflow engine

async def run_policy_step(step: Step, ctx: WorkflowContext) -> StepResult:
    results = []
    for policy_name in step.policies:
        policy = policy_registry.get(policy_name)
        result = policy.check(ctx)
        results.append(result)
        if not result.passed:
            log.warning(f"Policy {policy_name} FAILED: {result.reason}")
            # Policy failure blocks progression. No override.
            return StepResult(
                status="blocked",
                reason=f"Policy {policy_name}: {result.reason}",
                details=result.details
            )
    return StepResult(status="passed", details={r.name: r for r in results})

6. Eval System

Every eval step has explicit thresholds, bars, and criteria. Eval without thresholds is vibes. Eval results feed back into agent improvement over time.

6.1 Eval Agent Design

@dataclass
class EvalCriteria:
    name: str
    description: str
    threshold: float        # 0.0-1.0, score must exceed this
    weight: float           # relative importance
    rubric: list[str]       # specific items to check

@dataclass
class EvalResult:
    criteria: list[EvalCriteria]
    scores: dict[str, float]        # criterion_name -> score
    passed: bool                    # all weighted scores above thresholds
    reasoning: str                  # why (for audit trail)
    improvement_hints: list[str]    # for retry loops

6.3 Eval Feedback Loop

Eval results don't just gate workflow steps. They accumulate and feed back into agent improvement:

  1. Per-step scoring — Every eval step logs its scores to eval_history.jsonl
  2. Weekly aggregation — Memory Maintenance workflow computes per-agent pass rates
  3. Threshold adjustment — If an agent consistently scores >0.9, raise the bar. If <0.5, flag for skill gap.
  4. Skill gap → Skill Learning workflow — Identified gaps trigger skill creation or refinement

7.2 Memory Distribution

The Memory Organizer doesn't just compact — it actively routes feedback to where it's needed:

  • Writing feedback (voice, style corrections) → Writer agent skills
  • Publishing incidents (wrong content, bypassed policy) → Social agent skills + new policies
  • Research dead ends → Researcher agent "when to avoid" sections
  • Task routing errors → Orchestrator agent routing rules
  • User preferences → All agent prompt context

9. Human-in-the-Loop Design

Some workflows need your constant feedback. The workflow engine has a first-class "human checkpoint" step type that pauses execution and waits.

9.1 Checkpoint Mechanics

  1. Workflow reaches a step-human step
  2. Engine saves full state (all prior step outputs, current context)
  3. Sends notification: bridge item (iPhone) + WebGUI queue entry
  4. Workflow pauses. Other workflows continue. Engine is not blocked.
  5. Human reviews via WebGUI or iPhone: three actions available:
    • Approve — workflow resumes from next step
    • Reject — workflow loops back to specified retry step
    • Feedback — human provides text, workflow retries current step with feedback injected into agent context
  6. Timeout: if no human action within configurable window (default 24h), workflow auto-escalates or auto-proceeds based on step config

9.2 Which Workflows Need Human Checkpoints

WorkflowStepWhenDefault on timeout
Self-Evolution6. Human reviewStructural code changes (not minor fixes)Abort
Incident Response5. Human approvalCode changes (not restarts)Abort
Communication(dynamic)High-risk actions flagged by designerAbort
Article (optional)After quality evalIf user re-enables approval modeAuto-proceed
Design principle: Human checkpoints are for safety-critical decisions only. Content creation has full autonomy (your 2026-04-07 authorization). Add checkpoints where the cost of a mistake is high and irreversible.

10.2 Multi-Model Discussion Flow

  1. Topic enters the discussion space
  2. Each participant takes turns (round-robin or priority-based)
  3. Between full responses, agents emit chatty status updates (every 15-30 min)

12.2 Why Python, Not Rust

FactorPythonRust
BottleneckLLM API latency (1-10s). Python overhead is noise.Would save microseconds on a seconds-scale bottleneck.
EcosystemEvery LLM SDK, every API client, every data tool.Limited LLM ecosystem. Would write bindings for everything.
Iteration speedChange, run, see result. No compile step.Compile-debug cycle adds friction to rapid iteration.
Type safetydataclasses + Protocol + mypy get you 90% of the way.Full type safety, but at 5x development cost.
ReliabilityWorkflows + policies + tests give reliability at the system level.Language-level safety doesn't prevent LLM hallucination or API failures.

Verdict: Python with strict typing (mypy --strict), comprehensive tests, and clear module boundaries. The workflow engine provides the reliability that matters (checkpointing, retry, isolation). Language-level safety is the wrong layer to optimize.

12.3 Production Standards

  • Type checking: mypy --strict on all modules. No Any types except at LLM API boundaries.
  • Testing: All policies have unit tests. All workflows have integration tests with mock APIs. Coverage target: 80% for policies, 60% for agents.
  • Logging: Structured JSON logging. Per-step trace IDs. Log at boundaries and decisions, never log secrets.
  • Error handling: Errors propagate to workflow engine. Per-step retry with backoff. No silent swallowing.
  • Dependencies: Pin all versions. Audit for vulnerabilities quarterly. Minimal dependency surface.

Every workflow step has a typed input and output. No passing unstructured dicts between steps. This catches integration bugs at definition time, not runtime.

15. What You Might Still Be Missing

15.1 Workflow Versioning

When you update a workflow definition, running instances should complete on the old version. New instances use the new version. Each workflow carries a version: int field. The engine loads the version that was active when the workflow was triggered.

15.2 Graceful Degradation

What happens when an external API is down?

IntegrationFallback
Substack API downQueue article for retry. Social posting continues.
Twitter API downSkip tweet, log. Don't block article publish.
LLM API downQueue all LLM-dependent steps. Deterministic steps continue. Offline message to bridge.
Oura API downSkip health ingest for the day. No alert.
Postgres downFall back to SQLite for memory. Log degraded status.

15.3 Workflow Templates (Reusable Patterns)

Several workflows share a common pattern. Extract as templates:

  • Fetch → Process → Verify → Deliver: Used by Briefing, Market, Health, Growth Report
  • Create → Eval → Retry-or-Proceed → Publish: Used by Article, Podcast, Book Notes
  • Monitor → Classify → React: Used by Social Reactive, System Health, Folder Monitor

15.4 Testing Infrastructure

Three tiers of tests:

TierScopeSpeedWhen
UnitIndividual policies, data contracts, deterministic stepsFast (<1s)Every commit
IntegrationFull workflow with mock LLM + mock APIsMedium (<30s)Every PR
E2EFull workflow with real APIs (gated by cost)Slow (<5min)Weekly or manual

15.5 Rollback & Recovery

Every checkpoint saves full state. If a step fails after max retries:

  1. Engine marks workflow as failed_at_step_N
  2. Admin can inspect state at any prior checkpoint via WebGUI
  3. Admin can resume from any checkpoint, or abort
  4. For critical workflows: auto-escalate to human (bridge + WebGUI notification)

15.6 Concurrency & Resource Limits

  • Max concurrent workflows: configurable (default 5)
  • Max concurrent LLM calls: configurable per model (Opus: 2, Sonnet: 4, Haiku: 8)
  • TTS: sequential only (hard policy, enforced by engine)
  • Workflow steps within a parallel group run concurrently up to the LLM limit

15.7 Multi-User Considerations

The iOS app supports multi-user. Workflows that are per-user (Communication, Health) carry a user_id context. Shared workflows (System Health, Memory Maintenance) are global. The engine routes per-user workflows to user-specific state.

15.8 Deployment Safety

When self-evolution deploys code changes:

  1. Git commit on a branch (never direct to main)
  2. Run test suite (unit + integration)
  3. If tests pass and change is minor: auto-merge
  4. If tests fail or change is structural: human checkpoint
  5. After merge: engine reloads affected workflows without restart

16. Migration Path: V1/V2 → V3

Don't rewrite everything at once. Incremental migration, one workflow at a time, with the engine running alongside the existing system.

Phase 1: Foundation (Week 1-2)

  • Build the workflow engine core: engine/workflow.py, executor.py, checkpoint.py
  • Build the policy framework: policies/ with test suite
  • Build data contracts for the first workflow
  • Migrate the simplest workflow: System Health Monitoring (fully deterministic, zero LLM, low risk)
  • Run V3 engine alongside V1 core.py. Both active.

Phase 2: Deterministic Workflows (Week 2-3)

  • Migrate Intelligence Briefing (mostly deterministic fetch + one LLM step)
  • Migrate Health & Wellness (local LLM only, isolated)
  • Build WebGUI workflow monitor panel
  • Validate: are checkpoints working? Are policies catching errors?

Phase 3: Content Workflows (Week 3-4)

  • Migrate Article Creation (highest value, most complex)
  • Migrate Podcast Production (decoupled via folder monitor)
  • Build WebGUI human review queue
  • Build HTML migration for agent prompts and skills

Phase 4: Social & Self-Management (Week 4-5)

  • Migrate Social Reactive and Social Proactive
  • Migrate Daily Journal, Weekly Reflection
  • Build Memory Organizer agent
  • Build eval feedback loop

Phase 5: Advanced (Week 5-6)

  • Build Designer agent for dynamic DAGs
  • Migrate Communication workflow (the hardest one)
  • Build Multi-Agent Discussion workflow + WebGUI chat viewer
  • Build Self-Evolution with closed-loop eval
  • Decommission V1 core.py super agent

Phase 6: Polish (Week 6-7)

  • Build remaining WebGUI panels (cost tracker, eval dashboard, agent roster)
  • Complete HTML migration for all content types
  • Build Incident Response and Skill Learning workflows
  • Performance tuning: token budgets, model assignments, concurrency limits
  • Full test suite: unit + integration + E2E
Key constraint: At every phase, Mira must remain operational. The V3 engine runs alongside V1. Workflows are migrated one at a time. V1 handles everything not yet migrated. Only decommission V1 when all workflows are on V3 and stable for 1 week.
