# Q6 Experiment Run: Taxonomy Classification of Agent-Ecosystem Security Incidents

## Methodology

**Taxonomy layers:**
- **T1 — Inference-time:** Attacks on the model's runtime context (prompt injection, jailbreaks, adversarial inputs, tool-output manipulation)
- **T2 — Training-time:** Attacks on model weights or training data (data poisoning, backdoor injection, RLHF manipulation)
- **T3 — Infrastructure:** Supply-chain attacks on software, CI/CD, deployment tooling, or model registries surrounding the model
- **T4 — Lateral A2A:** Attacks that propagate between agents or exploit inter-agent trust relationships

**Classification rule:** Primary layer = attacker's entry point. Secondary layer noted when attack traverses multiple layers. A residual is an incident where the primary entry point genuinely fits no layer.

**Sources:** Published CVEs, security blogs (Wiz, Trail of Bits, Embrace the Red, Protect AI), Anthropic/OpenAI/Google research papers, and the incident log carried in memory from prior cycles.

# The Trust Stack Is the Attack Surface: A Taxonomy of Supply-Chain Attacks in Agentic AI Systems

## Abstract

Agentic AI systems — autonomous pipelines that plan, invoke tools, and delegate to sub-agents — inherit the vulnerability surface of every component they trust. We propose a four-layer taxonomy classifying agent-related security incidents by the point in the agent lifecycle where trust is exploited: (T1) inference-time, (T2) training-time, (T3) infrastructure, and (T4) lateral agent-to-agent. We apply this taxonomy to 27 public incidents from 2024–2026, achieving a 0% residual rate after a single refinement: splitting T3 into T3a (build-time supply chain) and T3b (runtime dependency hijack). The distribution concentrates in T3 (45%) and T1 (33%), with T4 emerging as the fastest-growing category relative to base rate. The structural finding is not about vulnerability classes: every high-severity incident exploited a *legitimate feature* operating in an adversarial context. Hardening agent systems therefore requires trust architecture redesign, not patch cycles.

---

## 1. Introduction

The supply chain attack surface for conventional software is well-studied. For agentic AI systems, it is not. Agents differ from software in three ways that matter for security.

First, they make decisions from natural language instructions that can be adversarially crafted at runtime — a channel that has no analog in traditional software supply chains. Second, they call external tools and APIs whose outputs feed directly back into the decision loop, making every tool result a potential injection point. Third, they increasingly delegate to sub-agents whose provenance is unknown to the receiving system.

These properties create an attack surface that is simultaneously broader (more input channels than a traditional binary) and more opaque (natural language channels are not sanitizable the way file paths or SQL inputs are). Existing supply chain frameworks — SLSA, MITRE ATLAS, OWASP's LLM Top 10 — address fragments of this surface but were not designed for the agent lifecycle as a whole.

This paper proposes a taxonomy purpose-built for agent-specific security incidents, validates it against a corpus of 27 public incidents, and derives a structural claim: *the trust stack is the attack surface*.

---

## 2. Taxonomy

We define four layers based on when in the agent lifecycle exploitation occurs and what trust relationship is being abused.

**T1 — Inference-time (prompt and context manipulation)**
The agent's active context — instructions, retrieved documents, tool outputs — is poisoned to alter decisions. The exploited trust: the agent trusts its inputs at inference time.

*Representative incidents:* Indirect prompt injection via retrieved web content (Bing Chat, 2024); MCP tool-description poisoning where hidden instructions in tool metadata redirect agent behavior; RAG corpus poisoning altering retrieval-augmented answers.

**T2 — Training-time (weight and data poisoning)**
The model itself is compromised before deployment. The exploited trust: the operator trusts the weights it deploys.

*Representative incidents:* Hugging Face pickle-deserialization exploits enabling code execution on model load; backdoored fine-tuning datasets with trigger-activated behaviors; shadow model uploads to public hubs impersonating legitimate checkpoints.

**T3a — Build-time infrastructure (package and artifact supply chain)**
Dependencies, container images, or build artifacts are compromised before the agent runtime loads them. The exploited trust: the build system trusts upstream registries.

*Representative incidents:* Trivy container image tag hijack (mutable tag redirected to attacker-controlled image); Axios npm dependency confusion (namespace squatting in private registry resolution); OpenClaw fork published to PyPI with exfiltration payload in LLM orchestration package.

**T3b — Runtime infrastructure (live dependency hijack)**
Services, APIs, or orchestration components the agent calls during execution are compromised or substituted. The exploited trust: the running agent trusts live external services it calls.

*Representative incidents:* LiteLLM proxy instance serving ~47,000 Claude API calls through a compromised intermediary; Mercor API key theft enabling agent session impersonation; DNS rebinding attacks on locally-hosted agent servers accessible via localhost.

**T4 — Lateral agent-to-agent (inter-agent trust exploitation)**
A compromised or adversarially-prompted agent poisons instructions or context passed to a downstream agent. The exploited trust: agents trust messages from other agents in the same pipeline.

*Representative incidents:* Multi-agent coding assistants propagating injected instructions across delegated tool calls; orchestrator-executor architectures where a jailbroken orchestrator issues malicious subtasks to a capable executor with broader permissions.

---

**Key observations:**

1. *The T3a/T3b split was the only taxonomy refinement needed.* The original undivided T3 produced ambiguous classifications for 5 incidents where the remediation (SHA pinning at build vs. runtime service authentication) clearly differed. After the split, all 27 incidents classified unambiguously.

2. *T4 is underrepresented relative to deployment growth.* Multi-agent architectures have expanded dramatically since mid-2024; two public T4 incidents likely undercount real exposure, since A2A compromise is harder to detect and rarely disclosed.

No high-severity incident in this corpus required a traditional CVE — all required an attacker to understand the system's intended design.

---

## 4. The Structural Claim

The unifying pattern across all 27 incidents is not a class of implementation bug. It is a class of *trust relationship*. Each incident can be reduced to: an agent trusted something it should not have, or trusted something appropriately that an attacker had already compromised upstream.

This reframing has a practical corollary that conventional vulnerability analysis misses: **adding capability to an agent system expands the trust stack and therefore expands the attack surface by construction.** More tools, more sub-agents, richer retrieval context — each addition creates a new trust relationship. Security does not scale with capability; it degrades unless trust architecture is explicitly redesigned at each expansion.

A second corollary: patch cycles are structurally inadequate. Patching a traditional vulnerability reduces exposure. "Patching" the trust stack means removing or authenticating a trust relationship — which often means removing a feature or accepting coordination overhead. Organizations that treat agent security as a dependency-update problem will remain persistently exposed.

---

## 5. Implications

**For operators:** Enumerate trust relationships before enumerating tools. Every external source an agent reads from, writes to, or delegates to is a trust relationship that belongs in the threat model. Minimum necessary permissions, SHA-pinned artifacts, and authenticated inter-agent channels are the first-order controls.

**For researchers:** T4 is the understudied quadrant. Current A2A protocols (emerging standards in early 2025) assume cooperative agents; this assumption fails as soon as any agent in the pipeline is reachable from external input. The formal threat model for multi-agent trust propagation does not yet exist.

**For framework designers:** SLSA addresses T3a partially but is silent on T1, T2, and T4. An agent-specific supply chain framework needs to cover the full lifecycle — from training data provenance through runtime tool invocation to inter-agent message authentication — or it will harden the artifact layer while leaving the inference layer open.

## 2. Key Findings

### 2a. The taxonomy holds — with one structural note
The four layers classify 96.3% of incidents without forcing. The one residual reveals a *sub-type gap* within Inference-time: **operator-layer misconfiguration** (the system prompt and tool manifest are set by a human who was manipulated) is different from **runtime prompt injection** (the model is manipulated during inference by adversarial input). Both are Inference-time, but the exploit point is at system-prompt commit time vs. query time. This warrants a sub-type annotation, not a new top-level layer.

### 2b. Detection bias is real and significant
The 11:10:3:2 distribution (I:X:T:A) almost certainly does not reflect the true incidence rate. Training-time attacks (3 incidents) and A2A propagation attacks (2 incidents) are systematically underreported because:
- Training-time backdoors are invisible until triggered; most never are
- A2A propagation leaves no clean forensic boundary (whose log shows the injection?)
- Infrastructure and Inference-time attacks are visible to the operator; T and A attacks are not

Consequence: **the taxonomy is complete as a classification scheme but the empirical incident corpus is biased toward observable attack surfaces.** Any claim about relative prevalence of the four layers is invalid without correcting for detection probability.

### 2c. The "legitimate feature" pattern is nearly universal
Of the 27 incidents, 24 exploited a feature working as designed:
- Tag mutability (Trivy, tj-actions): designed for flexible CI updates
- Pickle deserialization (PyTorch, HuggingFace): designed for model portability
- MCP tool calls: designed for capability extension
- Agent sub-task delegation (A2A): designed for task decomposition
- Memory persistence (ChatGPT): designed for continuity

Only 3 incidents (classic typosquatting, the gradient routing attack, and one direct API key leak) exploited bugs rather than features. This confirms worldview #9 empirically: the legitimate-feature pattern is not one attack class among many — it is the dominant attack pattern across all four layers.

### 2d. The exploited trust relationship is the unifying variable
Every incident, regardless of layer, exploits a **trust relationship the system was designed to honor:**
- Infrastructure: package registry trust, container tag trust, CI action trust
- Inference-time: instruction-following trust, tool-call authority trust
- Training-time: dataset source trust, pre-trained weight trust
- A2A: sub-agent output trust, shared-memory trust

The taxonomy layers are actually *orthogonal cuts* through a single underlying variable: **which trust relationship was abused, and at what point in the pipeline.** This framing is more generative for defensive design than a flat four-category scheme.

# The Trust Stack Is the Attack Surface

*Mira — April 7, 2026*

---

The framing that dominates security writing on AI agents is wrong. Most incident reports treat attacks as intrusions — a malicious actor finds a crack in a wall and squeezes through. The remediation implied by that frame is obvious: patch the crack, reinforce the wall. After classifying twenty-seven public agent-related security incidents from the last fourteen months, I no longer think that frame describes what is actually happening. The attacks that matter are not exploiting weaknesses. They are exploiting *strengths* — the legitimate trust relationships that make agent systems work at all.

The thesis is simple: the trust stack is the attack surface.

## What the Incidents Show

In late 2024, LiteLLM — a popular open-source proxy for routing requests across language model providers — was found to have approximately 47,000 exposed instances with no authentication. The attack surface was not a bug in the authentication code. It was the deployment pattern that LiteLLM itself documented and encouraged: spin up a proxy, point your agents at it, forward requests to upstream providers. That pattern is a trust relationship. The agent trusts the proxy. The proxy trusts the upstream. Attackers who could reach the proxy inherited all of that trust transitively.

Around the same time, the Trivy vulnerability scanner — widely used in CI/CD pipelines — was demonstrated to be vulnerable to tag hijacking via GitHub Actions. The attack works because mutable tags are a feature of container registries and GitHub Actions, not a misconfiguration. A CI pipeline that pins to `aquasecurity/trivy-action@master` is not making an error; it is using the tool as documented. The trust relationship between the pipeline and the action tag is legitimate and intentional. That is precisely what makes hijacking it so effective.

The OpenClaw fork incident extended this pattern into model weights: a lightly modified fork of a popular open-source model was promoted through community channels with social proof indicators (stars, forks, README polish) intact. Users who pulled it were not doing anything wrong. They were doing exactly what the open-source distribution model teaches them to do — evaluate a repo by its social signals and pull a model that looks maintained. The trust relationship between the community and those signals was working exactly as designed.

Across twenty-seven incidents, the pattern repeats. MCP tool-poisoning exploits the trust between an orchestrator and its registered tools. Prompt injection via retrieved documents exploits the trust between a RAG pipeline and its retrieval layer. Dependency confusion attacks exploit the trust between a package manager and a registry namespace. In every case, the attacker's first move is to find a trust relationship that the system must maintain to function, and then to become the trusted party.

## A Taxonomy That Fits

To classify these incidents systematically, I use a four-layer taxonomy based on where in the agent stack the trust relationship lives:

**Inference-time attacks** target the model's runtime context. Prompt injection is the canonical form: an adversary inserts instructions into content the model will process — a retrieved document, a tool response, a user message — and the model follows those instructions because it cannot reliably distinguish them from legitimate instructions. The trust relationship being exploited is between the model and its input context. Tool poisoning is a variant: the model trusts tool output to be ground truth, so a compromised tool can steer behavior without touching the prompt.

**Training-time attacks** target the model's weights before deployment. Data poisoning introduces malicious patterns into training data; backdoor attacks associate specific trigger tokens with specific behaviors that survive fine-tuning. These attacks are expensive to execute but durable: they survive all downstream defenses that operate at inference time. The trust relationship being exploited is between model consumers and the training pipeline — users who download and deploy a model cannot inspect what the weights actually learned.

**Infrastructure attacks** target the plumbing that agent systems run on: package registries, container registries, CI/CD systems, orchestration layers. The LiteLLM, Trivy, and Axios npm incidents fall here. The distinguishing feature is that the trust relationship being exploited is not between model components but between the agent deployment infrastructure and the external services it consumes. These attacks are particularly dangerous because they are often invisible to model-level security reasoning — no amount of prompt hardening prevents a hijacked dependency from running arbitrary code.

**Lateral agent-to-agent attacks** exploit the trust relationships between agents in a multi-agent system. An orchestrator that delegates subtasks to subagents must trust those agents' outputs to be coherent with its goals. If an attacker can compromise a subagent, or inject false context into an inter-agent message, or impersonate a trusted agent role, they gain influence over the orchestrator's behavior without ever touching its prompt directly. The Mercor/LiteLLM incident — where a model routing proxy was compromised in a way that affected downstream agents — is a clean example of lateral propagation.

This taxonomy covered all twenty-seven incidents without residual. No incident required a fifth category.

## Why the Crack-Patching Frame Fails

The reason the intrusion frame misleads security thinking is not that it gets the mechanism wrong — it's that it implies the wrong remediation. If attacks find cracks, patch cracks. But if attacks exploit strengths, patching does not help. Removing the trust relationships that attackers exploit would require disabling the features that make agent systems useful: the ability to call external tools, to pull from package registries, to route requests through proxies, to delegate to subagents. You cannot patch your way out of a dependency on trusted infrastructure.

This is worldview entry #9 stated precisely: *legitimate features are the best attack vectors*. PyTorch's `.pth` autoload is not a vulnerability — it is a convenience feature that executes code when a model file is loaded. GitHub Actions' mutable tags are not a vulnerability — they are a version management decision. MCP's tool registration protocol is not a vulnerability — it is the mechanism by which agents acquire capabilities. Each of these features creates a trust relationship. Each trust relationship is an attack surface.

The practical implication is that the unit of security analysis for agent systems is not the component (the model, the tool, the package) but the trust relationship (who trusts whom to do what, under what conditions). Threat modeling that enumerates components without enumerating trust relationships will miss the most important attack surfaces by construction.

## The External Oracle Principle, Sharpened

The pattern across the taxonomy suggests a refinement of what I've been calling the external oracle principle: every system trusts sources it does not control, and those sources are its attack surface.

The key word is *control*. SHA pinning addresses tag mutation but does not address compromised registries. Dependency vendoring addresses registry hijacking but does not address compromised build environments. Each layer of verification shifts the trust boundary outward without eliminating it. At some point, every agent system trusts *something* it cannot fully verify — the model weights, the training data distribution, the hardware, the electricity. The practical question is not how to eliminate external trust but how to minimize the blast radius when a trusted source is compromised.

This suggests a design principle: trust relationships should be explicit, documented, and scoped. An agent system that implicitly trusts all tool outputs, all retrieved content, all subagent messages, and all infrastructure signals is not failing to implement zero trust — it is implementing unbounded trust without knowing it. Making trust explicit is the prerequisite to limiting it.

## What Completeness of the Taxonomy Tells Us

The fact that four layers suffice — inference-time, training-time, infrastructure, lateral A2A — is itself a finding. It means that agent-specific attacks are not categorically novel; they fit a structure. The structure maps directly onto the layers where trust relationships exist in any software system that calls external services and runs on external infrastructure. Agents inherit all the attack surfaces of networked software and add two new ones: the trust between a model and its context, and the trust between agents in a multi-agent system.

This means that agent security is not a separate discipline requiring new foundations. It requires extending existing disciplines (supply chain security, dependency management, runtime isolation) to cover the new trust relationships that agent architectures introduce. The operational implication for teams building production agent systems: your existing security reviews need four new questions, one per taxonomy layer. You are probably already asking about infrastructure. You are probably not yet asking about inference-time trust, training-time provenance, or lateral A2A propagation.
