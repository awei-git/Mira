# Cross-Domain Synthesis

**Tags:** explorer, synthesis, creativity, analogy, knowledge

## Core Principle
Actively search for structural analogies between unrelated fields — strip domain-specific language to find transferable mechanisms, then apply the source domain's solutions to the target domain's problems.

## The Mechanism-Extraction Move
After processing content from any domain, ask: "What mechanism is this?" Strip domain-specific language and state the underlying structure.
- "Competitive moat" → "barrier to entry" → "asymmetric reproduction cost"
- "Viral growth" → "positive feedback loop with low marginal cost"

Once abstracted, search for the same structure in other domains.

## Mechanism Library (maintain and grow)
A catalog of transferable structures to tag new items against:
- Network effects, threshold/tipping dynamics, principal-agent problems
- Arms races, regression to the mean, option value
- Batch processing vs. streaming, redundancy vs. efficiency tradeoffs

## The Juxtaposition Practice
Deliberately juxtapose two unrelated items from the same reading session. Ask: do these share a deep structure? If yes, what does Field A's solution tell Field B about its problem?

## In Briefings
Include at least one cross-domain bridge per briefing: "This finding in [field] structurally resembles [concept in different field] — which suggests [implication]."

## Validation Test
A good cross-domain transfer specifies *which parts* of the new domain map and *which don't*. A bad transfer just uses the surface metaphor without mechanism.

## Source
